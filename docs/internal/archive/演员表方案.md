# 演员表 / Cast 方案

> 状态：**已确认**  
> 日期：2026-06-23  
> 确认日期：2026-06-23  
> 触发：NexusM 对照调研

---

## 决策摘要

| 决策项 | 结论 |
|--------|------|
| 头像类型 | anime → 角色图（Bangumi characters），tv/movie → 演员照片（TMDB credits） |
| 展示形式 | 横向滚动圆形头像列表，头像 48px + 角色名 + 声优/演员名 |
| 演员详情页 | 初期不做，演员不可点击 |
| crew（导演等） | 不做，只取 cast |
| 头像缓存 | 强制本地缓存，存 `cache_dir()/cast/` |
| 刷新策略 | 元数据刷新时连带刷新 cast |
| 空白态 | 演员表为空时整个区域不渲染 |

---

## 1. 背景

mochi 目前元数据覆盖：标题、年份、简介、类型/标签、评分、海报、横幅。演员/声优信息是明显的空白。NexusM 和市面主流媒体服务器（Plex、Jellyfin、Emby）均具备完整的演员表展示和演员详情页。

本方案讨论 mochi 如何补齐演员表能力，暂不涉及具体 UI 细节和前端交互设计——那些留待后续阶段。

---

## 2. 现状

### 2.1 已有能力

| 组件 | 状态 |
|------|------|
| TMDB API 连通 | ✅ `TmdbClient` 已实现 search / detail |
| Bangumi API 连通 | ✅ `BangumiClient` 已实现 search / detail |
| 双源路由策略 | ✅ anime → Bangumi 优先；tv/movie → TMDB 优先 |
| 图片缓存 | ✅ `download_image()` + 本地缓存路径 |
| SQLite 元数据存储 | ✅ `series` 表已有 bangumi_id / tmdb_id |
| 前端详情页 | ✅ 展示标题、年份、简介、类型、评分 |

### 2.2 缺失

- **TMDB credits 端点未调用**：`/movie/{id}/credits`、`/tv/{id}/credits` 均未接入
- **Bangumi characters 端点未调用**：`/v0/subjects/{id}/characters` 未接入
- **MetadataResult 无演员字段**：返回结构不含 cast
- **数据库无演员表**：没有 cast / person 相关表
- **前端无演员展示**：详情页没有演员区域

## 3. 数据源分析

### 3.1 TMDB Credits（真人影视）

**端点**：
- `GET /movie/{id}/credits` — 电影演职员
- `GET /tv/{id}/credits` — 电视剧演职员

**返回结构**（精简）：
```json
{
  "cast": [
    {
      "id": 123,
      "name": "演员名",
      "character": "角色名",
      "profile_path": "/abc.jpg",
      "order": 0,
      "known_for_department": "Acting"
    }
  ],
  "crew": [
    {
      "id": 456,
      "name": "导演名",
      "job": "Director",
      "department": "Directing",
      "profile_path": "/def.jpg"
    }
  ]
}
```

**关键字段**：`id`（person ID，可用于后续获取演员详情）、`name`、`character`、`profile_path`、`order`（排序）。

**费用**：免费，和现有 search/detail 共用同一个 API key。无需额外申请。

**头像尺寸**：和 poster 一样，用 `w185` 或 `w342`（演员头像不需要太大）。

### 3.2 Bangumi Characters（动画）

**端点**：
- `GET /v0/subjects/{id}/characters` — 条目角色列表

**返回结构**（精简）：
```json
[
  {
    "id": 789,
    "name": "角色名",
    "images": {
      "large": "https://..."
    },
    "actors": [
      {
        "id": 101,
        "name": "声优名",
        "images": {
          "large": "https://..."
        }
      }
    ]
  }
]
```

**关键差异**：
- Bangumi 的体系是 **角色 → 声优**，而不是直接的「演员 → 角色」
- 同一个角色可能有多个声优（不同版本）
- 一个声优可能扮演多个角色

**费用**：Bangumi API 免费，无 key 要求。

**头像可用性**：Bangumi 角色图和声优图覆盖不如 TMDB 完整，很多条目没有角色/声优头像。

### 3.3 TMDB Person Detail（可选扩展）

**端点**：`GET /person/{id}`

返回演员/导演的个人资料：biography、birthday、place_of_birth、known_for_department、也参演的作品列表。

这是 NexusM 演员详情页的数据来源。mochi 初期可暂不接入，先把 cast 列表做出来。

## 4. 数据模型设计

### 4.1 新增数据表

```sql
-- 人员表（跨 series 共享，一个演员出现在多部作品里只存一条）
CREATE TABLE IF NOT EXISTS person (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    source      TEXT    NOT NULL,  -- "tmdb" 或 "bangumi"
    source_id   INTEGER NOT NULL,  -- TMDB person_id 或 Bangumi character/actor id
    name        TEXT    NOT NULL,
    role_name   TEXT,              -- 角色名（TMDB 的 character 或 Bangumi 的角色名）
    image_url   TEXT,              -- 原始图片 URL
    image_cache TEXT,              -- 本地缓存路径
    is_actor    INTEGER DEFAULT 1, -- 1=演员/声优, 0=导演/其他 crew
    extra       TEXT,              -- JSON: 声优/角色关系、crew job 等
    UNIQUE(source, source_id, role_name)
);

-- 作品-人员关联表
CREATE TABLE IF NOT EXISTS series_cast (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    series_id   INTEGER NOT NULL,
    person_id   INTEGER NOT NULL,
    sort_order  INTEGER DEFAULT 0,
    FOREIGN KEY (series_id) REFERENCES series(id),
    FOREIGN KEY (person_id) REFERENCES person(id),
    UNIQUE(series_id, person_id)
);
```

**设计考量**：
- person 和 series 多对多，所以用关联表。
- `UNIQUE(source, source_id, role_name)` 允许同一个演员在不同作品中扮演不同角色时各存一条。
- `extra` JSON 字段给 Bangumi 的声优-角色多层嵌套留弹性。
- 头像单独缓存（不在 MetadataResult 流程里），由拉取时异步下载。

### 4.2 MetadataResult 扩展

`MetadataResult` 本身不加 cast 字段。原因：

- cast 是列表数据，放在 MetadataResult 里会让这个结构膨胀（它目前是扁平字段）。
- cast 拉取和元数据拉取可以解耦：元数据先落库、展示，cast 异步补全。
- cast 需要单独的表来存，不是 SQLite 的一个列。

替代方案：新增一个 `CastResult` 类型，`fetch_cast()` 函数返回它，由 command 层写入 person 和 series_cast 表。

## 5. 拉取流程

### 5.1 触发时机

| 时机 | 行为 |
|------|------|
| 扫描新 series | 元数据落库后再拉 cast（异步，不阻塞界面展示） |
| 手动刷新元数据 | 同时刷新 cast |
| 查看详情页 | 如果 cast 已缓存则直接展示；如果未拉取则触发拉取 |

核心原则：**cast 是增值数据，不应阻塞基础元数据的展示**。海报和标题先出来，演员表可稍后补上。

### 5.2 拉取路由

沿用现有双源策略，但需要处理边界：

| series_type | 主源 | 说明 |
|-------------|------|------|
| anime | Bangumi characters | 声优信息；如果条目无角色数据，回退到 TMDB（少数动画在 TMDB 也有 cast） |
| tv / movie | TMDB credits | 演员表 |
| unknown | 按歧义裁决结果走对应源 | 如果用户选了 Bangumi 就用 characters，选了 TMDB 就用 credits |

**Bangumi 回退场景**：很多 Bangumi 条目没有角色数据（尤其冷门动画）。此时可以尝试用 TMDB 补 cast（如果它在 TMDB 有对应条目），但 TMDB 的动画 cast 通常是英文演员名，中文适配度差。

### 5.3 调用链路

```
前端请求 cast
  → Tauri command: get_series_cast(series_id)
    → 查 series_cast + person 表
      → 有数据 → 直接返回
      → 无数据 → 触发 fetch_cast(series_id)
        → 读 series 表的 bangumi_id / tmdb_id / series_type
        → 路由到对应 API
        → 写入 person + series_cast 表
        → 异步下载头像（批量并发，限 5 个）
        → 返回 cast 列表
```

### 5.4 头像下载

- TMDB 演员头像用 `w185`（小尺寸，头像不需要大图）。
- Bangumi 角色/声优头像用 `large`（Bangumi 只有这一个尺寸）。
- 下载策略：先到先得，下载失败静默降级（不展示头像），不阻塞整体流程。
- 缓存路径：沿用现有 `cache_dir()` 规范，子目录 `cast/`。

## 6. 前端展示（已确认）

### 6.1 详情页演员区域

位置：海报右侧下半区（与元数据上区共享海报右侧空间，中间渐隐细线分隔）。

展示形式：
- 横向滚动圆形头像列表，头像直径 48px，间距 14px
- 每头像：圆形图 + 主名字 11px + 副名字 10px muted
- anime：主名字 = 角色名，副名字 = 声优名
- tv/movie：主名字 = 演员名，副名字 = 角色名
- hover：头像 `scale(1.12)` + 暖金 glow
- 头像缺失：首字母占位圆（底色 `accent-dim`）
- 边缘渐变遮罩（提示可横滚）
- 演员表为空时整个区域不渲染，剧集区自然上移

### 6.2 演员详情页

暂不实现。演员头像不可点击。

### 6.3 声优与演员展示统一

前端展示层不做区分。虽然 Bangumi 返回「角色→声优」、TMDB 返回「演员→角色」，语义方向不同，但展示统一为「头像 + 主名字 + 副名字」的卡片。用户不需要感知底层数据源的差异。

## 7. 实施阶段建议

| 阶段 | 范围 | 预估工作量 |
|------|------|-----------|
| **Phase A** | TMDB credits 拉取 + 存储 + 基础前端展示（真人影视演员表） | 1-2 天 |
| **Phase B** | Bangumi characters 拉取 + 声优展示（动画） | 1 天 |
| **Phase C** | 演员详情页（TMDB person detail + 参演作品） | 1-2 天 |
| **Phase D** | 头像批量下载优化、搜索过滤、cast 数据刷新策略 | 1 天 |
