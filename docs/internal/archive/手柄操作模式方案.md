# Mochi — 手柄操作模式方案

> 2026-06-22 起草，基于与 Hanako 的讨论。同日深入讨论后决定**暂缓实施**，待实际用户需求出现后再启动。讨论纪要和关键洞察记录于第十节，供将来重启时参考。

---

## 一、目标

为 mochi 提供手柄（Xbox / PlayStation / Switch Pro）全程操作能力，覆盖浏览、选片、播放三大核心场景。用户在沙发上用手柄完成从打开应用到开始播放的全流程，不碰键鼠。

---

## 二、核心决策：并入，不独立

**结论：手柄模式作为 mochi 应用内的一个交互模式，不做独立版本。**

理由：

1. **数据层共享是硬需求。** 媒体库、播放进度、设置都在同一份 SQLite + config.json 里。独立版本要么做数据同步，要么让用户扫两次库。
2. **mochi 架构天然适合双模。** `GlobalBackground` 固定浮层、路由页面切换、OSC 独立浮层——手柄模式是全局交互层的切换，不推翻现有组件结构。
3. **参考系成熟。** Steam Big Picture、Plex HTPC、Netflix TV 版都是"同一应用，两种交互模式"。

独立版本的唯一合理场景是将来做 Android TV / Apple TV，那是平台变了，不是输入设备变了。

## 三、技术基础

### 3.1 输入层

两条路径可选：

| 路径 | 方案 | 兼容性 | 复杂度 |
|---|---|---|---|
| 原生 Gamepad API | `navigator.getGamepads()` | Windows WebView2 对 Xbox 手柄完整支持 | 低 |
| tauri-plugin-gamepad-api | Rust `gilrs` 库 polyfill | Xbox + DualShock + Switch Pro，跨平台 | 中 |

推荐先用原生 Gamepad API 验证，后续按需引入插件扩展兼容性。

### 3.2 核心 Hook：`useGamepad`

- 监听 `gamepadconnected` / `gamepaddisconnected` 事件
- 每帧 `requestAnimationFrame` 读 `navigator.getGamepads()`，做 button/axis 状态 diff
- 物理输入 → 逻辑动作映射：
  - D-pad / 左摇杆 → `Navigate(up/down/left/right)`
  - A（Xbox）/ ✕（PS）→ `Confirm`
  - B / ○ → `Back`
  - X / □ → `Secondary`
  - Y / △ → `ToggleMode`
  - L1/R1（肩键）→ `TabPrev` / `TabNext`
  - Start → `Menu`
- 按键重复延迟：方向键按住时 150ms 初始延迟 + 80ms 重复间隔，防止疯狂跳格

## 四、焦点系统

### 4.1 全局 `FocusContext`

```ts
interface FocusState {
  currentFocusId: string | null;
  focusHistory: string[];  // 用于 B 键返回
  enabled: boolean;        // 控制器模式开关
}
```

### 4.2 组件注册

每个可聚焦组件注册自己的 `focusId` 和四方向邻居：

```ts
useFocusable({
  focusId: "poster-3",
  neighbors: { left: "poster-2", right: "poster-4", up: null, down: null },
  onConfirm: () => navigate(`/series/${id}`),
});
```

### 4.3 视觉语言

- 焦点元素：暖金 accent 色光环（和选集 pill 高亮逻辑一致），`box-shadow` 扩散 8px
- 焦点转移：framer-motion `layoutId` 驱动光环位移动画，120ms spring
- 手柄模式下自动隐藏鼠标光标（`cursor: none` on body）

## 五、逐页面适配

### 5.1 PosterWall（首页海报墙）

| 操作 | 映射 | 说明 |
|---|---|---|
| 左右移动焦点 | D-pad 左/右 | 海报间移动，触发 snap 滚动到可视区 |
| 切换筛选标签 | L1 / R1 | 全部 / 继续 / 动漫 / 影视 之间切换 |
| 进入系列 | A | 导航到 `/series/:id`，推入 focusHistory |
| 打开设置 | Start | 打开 Settings 弹窗 |

焦点逻辑：当前行所有可见海报参与焦点链（按 DOM 顺序），方向键在同行内移动。到达边缘时滚动海报墙。

### 5.2 SeriesDetail（系列详情 + 剧集列表）

| 操作 | 映射 | 说明 |
|---|---|---|
| 上下左右移动 | D-pad | 剧集网格导航（grid 布局下按行列邻居计算） |
| 播放 | A | 导航到 `/play/:episodeId` |
| 返回首页 | B | `navigate(-1)`，弹出 focusHistory |
| 打开 ⋮ 菜单 | X | 刷新元数据 / 修改匹配等 |

焦点逻辑：`episodes` 数组 + flex-wrap 网格，根据当前焦点 index 和列数计算邻居。`SeriesDetail` 的信息区（海报、简介）不做焦点——简化导航。

### 5.3 VideoPlayer（播放页）

| 操作 | 映射 | 说明 |
|---|---|---|
| 暂停/播放 | A | 切换播放状态 |
| 快进/快退 | D-pad 左/右 | 每次 ±5s，长按加速 |
| 音量 | D-pad 上/下 | 每次 ±5% |
| 显示/隐藏 OSC | 任意按键（播放中） | OSC 浮现 3s 后自动淡出 |
| 返回详情 | B | 退出播放 |
| 选集面板 | X | 打开选集 panel，焦点进入 panel 网格 |
| 全屏切换 | Start | |

OSC 在控制器模式下始终显示当前焦点按钮的标签（如 `▶ 暂停  [A]`），消除"哪个键干什么"的记忆负担。

### 5.4 Settings（设置页）

**不强求完美适配。** 设置页的核心操作是文本输入（TMDB Key、代理地址）和复杂选择，这些天生适合键鼠。手柄模式下仅保证基础导航：方向键在 section 侧边栏和内容区间移动，A 选择/切换。文本输入框获得焦点时提示用户换键鼠。

## 六、模式切换

三种触发方式并存：

1. **手动切换**：设置 > 通用 > 启用控制器模式（开关）
2. **热插拔检测**：插上手柄弹出 toast「检测到手柄，要切换到控制器模式吗？[切换] [忽略]」
3. **键盘快捷键**：`Ctrl+Shift+G` 快速切换，方便调试

---

## 七、不做的事情

- **不做 10-foot UI（电视距离优化）**。那是独立 TV 版本的范畴，包含字体缩放、高对比度主题、更大的触控目标。桌面端的手柄操作不需要这些。
- **不做手柄打字**。文本输入场景交给键鼠。
- **不做触觉反馈（haptic）**。`tauri-plugin-gamepad-api` 不支持，且 mochi 的交互不需要震动反馈。

## 十、2026-06-22 讨论纪要与决策

### 10.1 决策：暂缓实施

经深入讨论，手柄操作模式**不进入当前开发计划**。待 mochi 积累一定用户量后，如有较多用户提出需求，再启动实施。

当前文档保留作为技术预案，将来重启时可跳过架构论证，直接进入实施。

### 10.2 受众定位修正

讨论中明确了手柄模式的实际受众：

- **Win 掌机群体**（ROG Ally、Steam Deck、Lenovo Legion Go、AyaNeo 等）：7-8" 屏幕 + 内置手柄，mochi 的封面浏览范式天然适合这类设备
- **Steam 大屏模式用户**：电脑接电视/投影，手柄是主输入设备
- 传统"沙发 + 键鼠"场景的实际需求可能低于预期：坐沙发上用无线键鼠操作桌面应用并不罕见

7" 1080p 掌机屏幕在 ~30cm 手持距离下，mochi 现有的 `vw/clamp` 响应式布局无需专门适配。真正需要 10-foot UI 的是底座电视输出场景（55"+ 3m 距离），但那属于独立 TV 版的范畴。

### 10.3 播放页 OSC 需要独立手柄 OSD

这是讨论中最关键的技术判断。当前 `OscBar.tsx` 的交互设计深度依赖鼠标：

| 当前 OSC 控件 | 交互方式 | 手柄兼容性 |
|---|---|---|
| 进度条拖拽 knob | `mousedown` → `mousemove` → `mouseup` 精确拖拽 | **不可行**。摇杆映射鼠标坐标拖进度条反人类 |
| 音量滑块 | 垂直拖拽 + hover 显示数值 | **不可行**。同样的拖拽问题 |
| 进度条 hover 变粗 | `hoverTrack` 状态控制 CSS | **不可行**。手柄没有 hover |
| 7 个按钮横排一行 | 鼠标任意点击，0 步跳转 | D-pad 最多需 6 步才能从最左走到最右 |
| ⋮ 菜单 / 📋 选集浮层 | 点击打开，点击外部关闭 | 可做，但需要焦点陷阱 |

**结论：鼠标 OSC 和手柄 OSD 是两套互不兼容的交互语法。** 鼠标的交互语法是**空间映射**（屏幕上任意坐标可瞬间跳转），手柄的交互语法是**序列导航**（方向键在有限节点间移动）。不应试图做一个"通用的" OSC 同时适配两者——那会让两个模式都变差。

正确做法是手柄模式下渲染独立的 `GamepadOSD` 组件（与 `OscBar` 平级），设计要点：
- 进度条改为**只读**，快进快退用 ±10s 跳跃 + 按住加速曲线
- 控件从横排改为**垂直分层**（播放控制 → 辅助功能 → 进度条 → 标题），减少方向键步数
- 每个按钮标注对应手柄键位
- 通过 `FocusContext.enabled` 切换渲染哪个组件

### 10.4 浏览页适配量不变

PosterWall（横滚海报 = 一维序列）和 SeriesDetail（剧集网格 = 二维网格）的适配方向与原始方案一致，无需推倒重来。真正的额外工作量集中在播放页。

### 10.5 复杂度重估

| 阶段 | 原预估 | 讨论后重估 | 说明 |
|---|---|---|---|
| Phase 1 | 地基 | 地基 | 不变 |
| Phase 2 | 低 | 低 | PosterWall 横滚天然适合 D-pad |
| Phase 3 | 低 | 低 | 标准网格导航 |
| Phase 4 | 中 | **中高** | 不是"给 OSC 加标注"，是设计和实现独立的手柄 OSD 组件 |
| Phase 5 | 低 | 低 | 不变 |

### 10.6 重启条件

将来如果出现以下信号，考虑启动手柄模式：
- GitHub Issues 中出现多个手柄/掌机/Steam Deck 相关需求
- mochi 在 Win 掌机社区（如 ROG Ally 贴吧、Steam Deck 中文社区）被自发讨论
- mochi 积累足够用户量后，通过投票或反馈渠道确认需求优先级

重启时，本节纪要可作为设计起点，跳过"要不要做独立版""鼠标 OSC 能不能适配手柄"等已论证问题的重复讨论。
