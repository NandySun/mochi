# 音轨/字幕切换修复

> ✅ **已关闭 · 2026-06-30**
>
> 实际修复路径与原诊断不符。原假设根因为 `libmpv-wrapper` `mpv_wrapper_command` 里的 `trim_matches('"')` bug，但实际上 mochi 当前的 `refreshTracks` 调用链（`getProperty` 子属性查询）完全不走该路径。重新诊断后发现真正根因是 mpv 启动时序：v0.3.2 时期 `loadfile` 后 track-list 填充晚于 800ms，refreshTracks 静默失败；**v0.3.3 改 `loadfile` args 位置（加 `-1` index 占位）+ 注入 `sub-fonts-dir` 时意外修复了时序**，800ms 内 track-list 已就绪。三个验证点（mpv 切轨 / OSC 勾选更新 / console 静默）均通过。

---

## 问题

播放页点击切换字幕轨或音轨后，UI 不更新（轨道列表显示旧状态），且切换行为不可靠。

## 根因

`libmpv-wrapper v0.1.1` 中 `mpv_node` 结构体的序列化逻辑在读取 `track-list` 属性时崩溃。`refreshTracks()` 调用链：

```
getProperty("track-list/count") → libmpv-wrapper 序列化 track-list 节点 → 崩溃
```

crash 发生在 Rust FFI 层，前端 `catch { }` 吞掉了错误，UI 保持旧轨道列表。

## 影响范围

| 功能 | 状态 |
|------|------|
| OSC 字幕轨切换 | 切换指令发出但 UI 不更新 |
| OSC 音轨切换 | 同上 |
| OSC 轨道菜单展示 | 初次打开播放页时可能正常（外挂字幕由 `loadSubtitleFiles` 注入），但后续切换后菜单不更新 |
| 选集面板轨道标记 | 间接影响 |

## 可修复路径

1. **绕过 `refreshTracks` 的 track-list 读取**（推荐先行）
   - 前端维护轨道列表状态，不给 libmpv-wrapper 读 `track-list/count`
   - 通过 `observe_property("sid")` + `observe_property("aid")` 监听当前选中轨
   - 外挂字幕轨由 `loadSubtitleFiles` 的路径列表推算

2. **Fork libmpv-wrapper 修复序列化**
   - 库体量极小（17 commits，核心是 FFI 序列化）
   - 上游已休眠 7 个月，无同步负担
   - 修复后需替换 DLL 并验证

## 当前状态

- libmpv-wrapper: v0.1.1（最新，2025-11-23）
- tauri-plugin-libmpv: 0.3.2（最新）
- 上游 issue: 无公开报告
- 优先级：中（不影响核心播放，但影响多轨内容的体验）

## 补充（v0.3.3 字体加载排查后，2026-06-30）

v0.3.3 字体加载排查时同步翻了一遍 libmpv-wrapper 源码，跟音轨问题相关的几条洞察：

### 根因细化

原描述"mpv_node 序列化崩溃"过于模糊。具体定位到 `libmpv-wrapper` 仓库的 `src/ffi.rs` `mpv_wrapper_command` 函数里这段：

```rust
_ => v.to_string().trim_matches('\"').to_string(),
```

这是处理 `serde_json::Value` 的 match 分支。`Bool`/`Number`/`String` 三个分支有专门处理，但 `Object`/`Array` 走这条兜底分支——

- `Value::Object.to_string()` 输出 `{"k":"v"}` 这种带花括号的标准 JSON
- `trim_matches('\"')` 只去首尾双引号，对花括号无效
- 结果把带 `{}` 和 `"` 的原始 JSON 字符串直接喂给 mpv 内部，mpv 在解析嵌套节点时 panic

而 `track-list` 恰好是嵌套结构（array of objects，每个 object 含 `type`/`id`/`title`/`selected` 等字段），所以一读就炸。

**`String` 透传路径是干净的**——`Value::String` 走 `s.clone()` 原样传递，v0.3.3 字体加载的 `sub-fonts-dir=...` 注入验证过。所以 `setSubTrack` / `setAudioTrack` 用 `command("set", ["sid", ...])` 不会触发这条 bug——**这部分绕开已经做到了**。

### 短期绕过方案的限制

原"可修复路径 1"建议前端维护轨道列表、不读 track-list。但实际有个尴尬点：

进入播放页**首次**要构建 sid/aid 列表时，仍然需要读一次 `track-list/count`——这一步**必然触发**同一个 bug。

完全绕开意味着 UI 只能显示"无"、"1"、"2"这种硬编码占位轨道 ID，对外挂字幕特别不准确（外挂字幕轨 ID 是动态分配的）。这是**对完整 UI 体验的降级**，不是真正的"绕过"。

### 建议优先级调整

原文档把"路径 1 绕过"标为"推荐先行"。v0.3.3 排查后建议反过来：

- **先做路径 2 fork 修复**，实际工作量更小、收益更彻底
  - 修复点就是 `ffi.rs` 那 1-2 行的 Object/Array 分支
  - 正确写法是递归序列化为 mpv JSON IPC 期望的 `{k:v}` / `[v1,v2]` 形式
  - 预计 diff < 30 行
  - 库体量 17 commits，clone 下来读 30 分钟就能找到改点

- 路径 1 的"前端维护轨道列表"改造可以**后置**——fork 修好之后仍有用（用 `observe_property("sid"/"aid")` 让 UI 实时反映当前轨，比现在全靠轮询更优雅），但不再是"为绕开 bug"的妥协

### Fork 修复方向

在 `libmpv-wrapper` `src/ffi.rs` `mpv_wrapper_command` 里，把 Object/Array 分支替换为正确序列化。两种写法可选：

**方案 A**：保留现有 args 数组结构，让递归序列化每项为 mpv 接受的字符串：

```rust
Value::Object(map) => {
    let pairs: Vec<String> = map.iter()
        .map(|(k, v)| format!("{}={}", k, serde_json_to_mpv_value_string(v)))
        .collect();
    pairs.join(",")
}
Value::Array(arr) => {
    arr.iter().map(serde_json_to_mpv_value_string).collect::<Vec<_>>().join(",")
}
```

**方案 B**：直接 JSON encode 整体（如果 mpv C API 接受 JSON object 字符串作为 options）：

```rust
Value::Object(_) | Value::Array(_) => v.to_string(),
```

但方案 B 跟原 bug 撞同一个雷（`to_string()` 已经输出了正确 JSON，trim 是多余的），去掉 trim 就够了：

```rust
_ => v.to_string(),  // 直接用 serde_json 的 to_string，输出标准 JSON
```

方案 B 改动最小（删 2 个字符 `.trim_matches('\"')`），但需要验证 mpv C API 接受嵌套 JSON 字符串作为 command argument。**大概率可行**——mpv 自己的 JSON IPC 协议就支持这种形式。

### 未来风险（不阻塞 v0.3.4）：`libmpv-wrapper` 的 trim bug

原根因诊断里定位的 `mpv_wrapper_command` 兜底分支：

```rust
_ => v.to_string().trim_matches('\"').to_string(),
```

这条 **bug 仍然真实存在**——但 mochi 当前所有 `command(...)` 调用的 args 都是 String 类型（透传路径），不触发该分支。**未来如果增加需要 Object/Array args 的命令调用**（例如 `sub-add` 带 `flags` 字典、`set` 命令的 list-options 等），会直接踩坑。

**建议**：在合适时机 fork `nini22P/libmpv-wrapper` 到 `NandySun/libmpv-wrapper` 并修这一行。许可证为 LGPL-2.1，动态链接到闭源应用天然合规，不会污染 mochi 自己的许可。改动量极小（1-2 行），但需要：
- 三个平台分别 build 出 .dll / .dylib / .so
- 替换 `src-tauri/lib/` 下对应文件
- 写 CI 脚本避免 macOS/Linux 落地时重新踩坑

**优先级**：低。**不是 v0.3.4 阻塞项**，记一笔留给将来。

### 补充：v0.3.3 同期暴露的 fonts_dir 回归 bug（已修）

v0.3.3 字体加载改动还引入了一个更严重的问题——`db::get_episode_fonts_dir` 的闭包类型推断在 nullable 列上错误地把 `row.get(0)` 推断成 `String` 而不是 `Option<String>`，导致 `series.fonts_dir` 是 NULL 时（v0.3.3 之前的所有 series 都是 NULL）报 `Invalid column type Null`。**所有没 fonts/ 目录的视频无法播放**。

修复（`src-tauri/src/db.rs`）：用 `let raw: Option<Option<String>> = ...` 显式标注 + `.map_err(|e| DbError::Sqlite(e))` 显式构造 + `raw.flatten()` 拍平嵌套 Option。三步踩坑链已记录，**v0.3.4 集成测试必须覆盖"旧库 + nullable 列"组合**。
