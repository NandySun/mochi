# 外挂字幕支持方案

> 2026-06-29 起草 · 实验素材：Cowboy Bebop（YakuboEncodes 视频 + TxxZ 字幕）

---

## 1. 现状评估

结论先行：**外挂字幕管道已建成 70%。** 扫描器收集、DB 存储、播放器加载三条链路都已布线，但有三处断点。

### 1.1 已就绪

| 层 | 做了什么 |
|---|---|
| 扫描器 | `SUB_EXTS` 常量（ass/ssa/srt/sub/vtt/idx）；`count_subtitles()` 按视频 stem 匹配同目录和 `sub/`/`subs/` 子目录的字幕文件 |
| 数据模型 | `episodes.subtitle_count` + `episodes.subtitle_paths`（JSON 数组）已建表、已迁移 |
| 后端 | `commands.rs` 的扫描入库流程序列化 `subtitle_paths` 为 JSON 写入 DB |
| 前端类型 | `Episode.subtitle_paths: string[]` |
| 播放器 | `VideoPlayer.tsx` 加载文件后调 `mpv.loadSubtitleFiles()`；`useMpv.ts` 用 `command("sub-add", [path])` 逐条注入 mpv |
| 选集刷新 | 加载外挂字幕后再等 800ms 二次 `refreshTracks()`，确保 mpv 的 track-list 已同步 |

### 1.2 两处断点

**断点 A：字幕匹配仅限同 stem，不支持异源命名**

`count_subtitles()` 的匹配逻辑：

```rust
let match_sub = |sub: &&PathBuf| -> bool {
    sub.file_stem().to_lowercase() == video_stem_lower   // 完全相等
    || sub.stem.starts_with(video_stem + ".")            // stem.xxx
    || sub.stem.starts_with(video_stem + "_")            // stem_xxx
};
```

对 Cowboy Bebop 的场景失效：
- 视频 stem：`[YakuboEncodes] Cowboy Bebop - 01 [BD 1080p 10bit][x265 HEVC][Dual Audio 5.1]`
- 字幕 stem：`[TxxZ&POPGO&MGRT][Cowboy_Bebop][BDrip][01][1440x1080_x264Hi10P_flac][0CB3724A]`

两者 stem 不共享任何前缀。唯一的交集是集号 `01`。

**断点 B：字体文件未接入 mpv**

扫描器没有 `FONT_EXTS` 常量，不会收集字体文件。mpv 初始化时未设 `sub-fonts-dir`，libass 找不到字幕引用的自定义字体时会回退系统默认字体，ASS 排版全毁。

---

## 2. 方案总则

参照 mochi 的 NFO 解析模式：**被动检测、零配置、不存在则零开销。**

不新增 UI 开关。不加字幕管理页面。mochi 不是字幕工具，只是把已经存在于文件系统里的字幕/字体文件准确地喂给 mpv。

---

## 3. 断点修复

### 3.1 集号近似匹配（断点 A）

**改动位置**：`scanner.rs` — `count_subtitles()` 函数

**方案**：在精确 stem 匹配无结果时，追加一层集号匹配作为兜底。

逻辑：
1. 先做现有精确 stem 匹配（保留，因为性能更好且优先）
2. 如果精确匹配挂了（count == 0），且视频文件已通过正则提取出了 `episode_number`，则对字幕文件列表做同样的正则提取
3. 字幕文件的 `episode_number` 与视频文件的 `episode_number` 一致 → 匹配

`count_subtitles()` 当前签名不接受 episode_number，需要调整：

```rust
fn count_subtitles(
    root_subs: &[PathBuf],
    sub_dir_files: &[PathBuf],
    video_stem_lower: &str,
    episode_number: i32,            // 新增：用于兜底匹配
) -> (i32, Vec<String>)
```

内部逻辑伪码：
```
1. exact_matches = match_by_stem(root_subs + sub_dir_files, video_stem_lower)
2. if exact_matches is not empty → return exact_matches  // 精确匹配优先
3. // 兜底：集号匹配
4. for each sub in (root_subs + sub_dir_files):
     extract_episode_info(sub.file_name)
     if episode_number matches → collect
5. return collected
```

**为什么要精确匹配优先？** 多版本字幕是常态（双语版 vs 纯中字 vs 特效版）。同 stem 命名的字幕应被优先识别，集号兜底只在"一个都没匹配到"时才触发，避免将远方的字幕错误关联。

**注意**：集号匹配需要排除已关联到其他视频的字幕（一个字幕文件不应被两集同时认领）。但考虑到当前 `count_subtitles()` 是每集独立调用，跨集去重不是硬需求。先做简单的：集号匹配到就收。

**测试素材**：Cowboy Bebop 原始状态（不改名），扫描后验证 Episode 01 的 `subtitle_paths` 是否包含 `[TxxZ&POPGO&MGRT][Cowboy_Bebop][BDrip][01]....ass`。

### 3.2 字体目录传入 mpv（断点 B）

**设计决定**：mochi 不在扫描器侧收集字体文件路径，不为字体建 DB 字段。字体的存在和位置是文件系统层面的约定，mochi 只负责在播放前把正确的路径传给 mpv。

**约定**：
- 字体文件（`.ttf` / `.otf` / `.woff` / `.woff2`）平铺在系列文件夹下的 `fonts/` 子目录
- 同时系列文件夹本身也作为字体搜索路径（兼容字体直接放在视频同目录的用法）
- mpv 的 `sub-fonts-dir` 接受分号分隔的多路径，两个目录一起传

**改动位置**：`useMpv.ts`、`VideoPlayer.tsx`

#### useMpv.ts：新增 setFontsDir

```typescript
const setFontsDir = useCallback(async (dirs: string) => {
  try {
    await setProperty("options/sub-fonts-dir", dirs);
  } catch { /* non-critical */ }
}, []);
```

#### VideoPlayer.tsx：loadFile 前拼接路径

```typescript
// 字体搜索路径：fonts/ 子目录 + 视频所在目录（mpv 多路径分号分隔）
const parentDir = filePath.replace(/[/\\][^/\\]+$/, "");
const fontsDir = parentDir + "/fonts";
mpv.setFontsDir(fontsDir + ";" + parentDir);
await mpv.loadFile(filePath);
```

不需要检查 `fonts/` 是否存在——mpv 遇到不存在的路径会静默跳过。

**为什么在 loadFile 之前设置？** `sub-fonts-dir` 影响 libass 的字体初始化，必须在视频文件加载前设定，libass 解析第一个 ASS 事件时就会用到。

**测试素材**：Cowboy Bebop 的 7 个字体文件放入 `Cowboy Bebop/fonts/`，字幕文件放入 `Cowboy Bebop/subs/`。播放时对比有字体/无字体的 ASS 排版差异。

## 4. 非目标（明确不做的）

- **字幕文件格式转换**（如 ass→srt）。交给 mpv/libass。
- **字幕下载/在线搜索**。mochi 是本地影库，不是字幕聚合器。
- **字体安装管理 UI**。字体只是文件，遵循 `fonts/` 目录约定即可。
- **字体压缩包解压**（`.rar` / `.zip` / `.7z`）。mochi 只消费 mpv 能直接读取的平铺字体文件，解压是用户的一次性前置操作。
- **字幕同步偏移调整 UI**。mpv 已有 `sub-delay` 快捷键（`[` / `]`），mochi 只保证字幕被正确加载即可。
- **内封字幕轨的提取/管理**。那是视频文件内部的事，mpv 原生处理。
- **字幕内容的解析或预览**。不解析 ASS 语法、不提取文本。
- **平面模式（flat mode）字幕收集**。平面模式使用场景极少，且用户想用外挂字幕时大概率已建系列文件夹。不在本次范围内。
- **Episode 级别的字体关联**。字体是系列范围的资源，不对齐单集。

## 7. 与 mochi 设计原则的对齐

- **零配置入库** ✓ — 有字幕就加载，没有就跳过，无开关
- **被动兼容** ✓ — 对标 NFO 解析模式：不制造、不管理、只消费
- **本地优先** ✓ — 字幕和字体都是本地文件，不涉及网络请求
- **减法哲学** ✓ — 不改 DB schema，不增 UI，纯粹补齐已有管道

## 8. 实施记录（2026-06-29）

### 已完成

| 改动 | 文件 | 说明 |
|------|------|------|
| 集号兜底匹配 | `scanner.rs` | `count_subtitles()` 新增 `episode_number` 参数，stem 匹配失败时回退到正则提取集号匹配 |
| 前端 JSON 解析 | `VideoPlayer.tsx` | `subtitle_paths` 从 Rust 传入是 JSON 字符串 `'["path"]'`，前端需 `JSON.parse` 为数组后再遍历 |

### 未做（暂不实施）

- 字体文件自动加载（`sub-fonts-dir`）：`setProperty("options/sub-fonts-dir", ...)` 在 libmpv-wrapper 中对 string 类型的属性设置会污染 mpv 句柄，需另行探索 |
- libmpv-wrapper 的 `track-list` 序列化崩溃修复：上游未修，见 `音轨切换修复待办.md` |

### 踩坑记录

1. **`subtitle_paths` 是 JSON 字符串而非数组**：Rust 侧 `episodes.subtitle_paths` 存的是 `Option<String>`（`serde_json::to_string(&Vec<String>)`），Tauri IPC 传到前端是 JS 字符串。直接用 `for...of` 会逐字符遍历，导致 mpv 被数十个单字符 `sub-add` 命令淹没。
2. **`setProperty` 传 string 类型会在 libmpv-wrapper v0.1.1 中污染 mpv 句柄**：所有其他 `setProperty` 调用均传 number（`time-pos`/`volume`/`speed`），传 string（`options/sub-fonts-dir`）导致后续所有 mpv 操作失效。
